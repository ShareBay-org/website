About

FreeList XML (or FLiX) is an XML file standard for sites and applications that provide free public resources like goods, services or information. Similar to a website’s sitemap, a FLiX file is an XML document that contains the application’s current inventory of free listings and is usually placed in the application’s public root folder, eg. ‘www.example.com/freelist.xml’

The purpose of FLiX is to create a standard method for sharing free listings data that participating sites and applications can contribute to and benefit from. For example, using FLiX, two unrelated sites can aggregate each other’s listing data, meaning that the users of both sites have a larger and more diverse inventory to choose from.

FLiX files are public files and can be used by anyone. You don’t need to be a contributor. Any third-party aggregator can use them to display items of interest to their audiences. The advantage to FLiX contributors is that they can reach a global audience more easily with backlinks to their site for users who wish to know more about their offers.

Implementation

If your site or application offers free items, services or knowledge, add your listings to an XML file called freelist.xml and place it in your site’s home directory. You can then share this link with anyone who you'd like to aggregate your listings into their site, or add it to our registry so other site owners can find it.*

Here’s the most basic example of an FLiX file with one offer:

<?xml version="1.0" encoding="utf-8" standalone="yes"?>
<freelist>
	<header>
		<appID>example.com</appID>
		<date>2022-07-12</date>
	</header>
	<list>
		<item>
			<title>A free item</title>
			<url>https://example.com/a-free-item.html</url>
		</item>
	</list>
</freelist>

Note that your appID must be a unique ANSI string. Your site’s domain is normally used. Check the registry to ensure you are using a unique appID and validate your XML before implementation. A file date is also required so aggregators can tell if they need to update their data or not.

* Registering your Freelist.xml file is not required, but doing so will greatly increase your chances of data syndication and reduce the chances of errors. Bear in mind some aggregators will only accept files with a registered appID.

Here’s a more complete implementation with one listing that makes use of all possible values:

<?xml version="1.0" encoding="utf-8" standalone="yes"?>
<freelist>
	<header>
		<appID>example.com</appID>
		<appName>Example Website</appName>
		<appDesc>A Website of Free Things</appDesc>
		<date>2022-07-12</date>
	</header>
	<list>
		<item>
			<title>A Free Bike</title>
			<url>https://example.com/a-free-item.html</url>
			<description>Red with go-faster stripes!</description>
			<imageURL>https://example.com/a-free-item.html</imageURL>
			<terms>Free</terms>
			<category>Bicycles</category>
			<city>Boston</city>
			<state>Massachusetts</state>
			<countryName>United States</countryName>
			<countryCode>US</countryCode>
			<lat>42.29988</lat>
			<long>-71.46388</long>
			<itemID>1322</itemID>
			<quantity>1</quantity>
			<tags>bike,bicycle,cycling</tags>
			<updated>2022-07-12</updated>
		</item>
	</list>
</freelist>

Complete list of header parameters

These parameters should appear once at the top of the file inside the <header></header> tag.

Parameter Required Data type Description
appID required string Your unique originator ID. Usually a domain name, but any string is acceptable as long as it’s unique. eg. “app_345”. Check FLiX registry to ensure you use a unique ID
date required yyyy-mm-dd The date when the file was created as a W3C date; eg.“2022-11-14”
appName optional string The name of the originating site; eg. “My Free Site”
appDesc optional string An optional description of your site/app; eg. “A website for cool, free stuff”

Complete list of item parameters

All your listings should be contained inside a <list></list> tag. Each listing then has its own <item></item> tag which contains the individual parameters specific to each listing.

Parameter Required Data type Description
title required string (max 100 bytes) The title of the listing; eg. “Boy’s Bicycle”
url required string URL The unique URL of the listing; eg. “”
type optional string What’s on offer. Should be ‘item’, ‘skill’, or ‘info’. Info should be used for any non-transactable listing; eg. A meeting place, or an announcement. (Readers: Where blank, malformed or missing, ‘item’ should be the assumed type)
quantity optional integer The number of items available. (Readers: Where blank, malformed or missing, 1 should be the assumed value)
terms optional string ‘free’, ‘loan’, ‘commons’, ‘swap’. (Readers: Where blank, malformed or missing, ‘free’ should be the assumed terms)
imageURL optional string URL The main image of the listing; eg. “https:// example.com /images/1.jpg”
description optional string (max 500 bytes) The listing description; eg. “Five speed trekking bike, hardly used.”
city optional string The city location of the listing; eg. “Boston”
state optional string The state or region of the listing; eg. “Massachusetts”
countryName optional string The name of the country of the listing; eg. “United States”
countryCode optional 2-letter ISO string The country code; eg. “US”
lat optional decimal The latitude coordinate of the listing; eg. 42.29988
long optional decimal The longitude coordinate of the listing; eg. -71.29988
itemID optional string Your unique listing ID for this item.
language optional ll-CC ISO language and locale code; eg. “en-US” (American English)
updated optional yyyy-mm-dd Date when the listing was last updated
category optional string Your category name for the listing; eg. “Bicycles”. Note: there is no universally-agreed format to listing categories, but we have compiled a list of the most common ones here that you might like to use in your application.
tags optional comma-separated string Keyword tags associated with the listing; eg. “Bicycles, Cycling, Racing Bikes”

Note: even though location data like city and lat/long coordinates are not required, they should be supplied wherever possible to ensure useful data portability. ie. Your listings may appear anywhere in the world and there’s not much point listing a free bicycle if no-one knows where it is!

Best practices for contributors

Before creating your file, please consider these best practices to ensure a good experience for all users of this data:

  1. Ensure you are using a unique appID which will uniquely identify your content on other networks. Check our registry to make sure your appID is unique, then add yours to ensure no-one else uses it;

  2. Follow the recommended listing parameters as much as you can to ensure a better experience for other networks and end-users, especially item location, image and description;

  3. The maximum file size is 10MB or 10,000 listings. If your site has more listings, consider using extra files (eg. freelist-1.xml, freelist-2.xml, etc), and create a file index; (find out how)

  4. Purge and rewrite your inventory file regularly according to how busy your site is. Once every 24 hours should suffice, but more often if your site has a high volume of transactions. Note: unlike RSS files, your FLiX file should contain all your current live offers, not just the latest.

  5. Please observe the correct data types and string length limitations.

  6. Do not include names, contact information of individuals or any sensitive data in your inventory, as you might breach data privacy rights;

  7. If you have content quality metrics available, consider only including higher quality listings to your inventory to help reduce spam and fake offers on other networks;

  8. Ensure that completed, expired or removed listings are removed from future inventory files to avoid confusion and broken links;

  9. Ensure that any image urls you set are accessible from any server;

  10. Compress larger files with Gzip to save bandwidth. (find out how)

  11. Remember to validate your XML before submission!

Best practices for readers & aggregators

Before including FLiX inventory files in your site or application, please observe these best practices:

  1. Always display the originator name or appID anywhere the originator’s listings are displayed;

  2. Always link to the originating site specified in the url string

  3. Reload the inventory and refresh your database often to avoid expired / broken links in the listings. Once every 24 hours is adequate in most cases. NB. Each inventory file is a complete inventory and should overwrite all previous data from the same originator;

  4. Be fussy about what data you import. Use filters to skip data that you might not want, or require certain parameters that you do. For example, if your site/app uses a map, make sure imported listings have a lat/long pair.

  5. Do not alter or amend the data included in the inventory file;

  6. Do not use or publish the data except for the purposes of displaying free listings;

  7. Do not use the data to obtain or publish any other data from the originator’s site (eg. Usernames or contact information);

Compressing your files with gzip

We recommend compressing larger files (>2MB) with Gzip to reduce bandwidth usage. XML files can be greatly compressed which means that your files can be sent and downloaded up to five times faster. Gzipped files have the extension .GZ, so we recommend that you output your XML and Gzip it to a file called freelist.xml.gz.

Reading a freelist.xml.gz file will require that you decompress it first.

Implementing Gzip varies from platform to platform, so we recommend you search ‘GZIP’ along with whatever platform / language you’re using to find out how to compress and decompress your files with Gzip.

Using multiple inventory files

If your site has more than 10,000 listings, or your XML file exceeds 10MB, you should split your listings into multiple files (eg. freelist1.xml, freelist2.xml, freelist3.xml, etc), and list these files in a separate index file.

Each file should be listed in its own <file></file> tag, nested inside an <index></index> tag, like this:

<?xml version="1.0" encoding="utf-8" standalone="yes"?>
<index>
	<file>https://example.com/freelist1.xml</file>
	<file>https://example.com/freelist2.xml</file>
	<file>https://example.com/freelist3.xml</file>
</index>

Note: file URLs should be absolute URLs, not relative.

We recommend naming the index file freelist.xml and numbering the individual files accordingly as per this example. You can then add this file into our registry as the main file.

When reading any freelist.xml file, test for the <index></index> tag first before trying to extract any listing data. If the index tag exists, then iterate through the listed files and extract data from each of them. If no index tag exists, then extract data from the main file as normal.
